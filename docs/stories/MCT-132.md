# MCT-132: Compactor stabilize Epic — mem_limit + observability + ParquetWriter close

## §1 배경

`mctrader-compactor` 컨테이너의 RAM 사용이 **48.5 GiB / 62.73 GiB (77.32%)**, CPU **122%** 까지 폭주 (단일 측정, 2026-05-11). 호스트 단일 Docker 인스턴스 (62.73 GiB 물리 메모리) 위에서 운영되어 호스트 OOM 직전 단계.

Trigger 가설: MCT-103 50 sym 확장 (forward-only history 누적 + L3 daily compaction accumulator) 이후 발현. 단 baseline (50 sym 확장 전) 기록 부재 + 추세 (증가/plateau) 미상 — 4 가설 (genuine leak / accumulator bloat / GC retention / legitimate scale ceiling) 중 어느 것도 사전 단정 불가.

상세: `docs/superpowers/specs/2026-05-11-compactor-memory-fix-design.md` §1.

## §2 목표 및 범위

Epic-A goal triad:
- **(a) Immediate OOM mitigation** — docker `mem_limit: 32g` + `memswap_limit: 32g` (swap disabled) 로 호스트 OOM 위험 즉시 차단
- **(b) Observability infra** — `/metrics` HTTP server (port 8080) + 5 Prometheus metrics + Grafana dashboard 로 Epic-B 진단 근거 확보
- **(c) Real leak candidate fix** — L1/L2/L3 ParquetWriter context manager wrap + tmp cleanup on exception + interval-driven `gc.collect()` (handle leak 가설 직접 fix)

Companion Epic: MCT-135 (planned, Epic-B v2 재설계 — streaming compaction + per-tier subprocess + ADR-017 amendment).

spec: `docs/superpowers/specs/2026-05-11-compactor-memory-fix-design.md` §2 (2-Epic 구조)

| 결정 | 선택 | 근거 |
|------|------|------|
| mem_limit value | 32 GiB (호스트 62.73 GiB 의 절반) | 호스트 OOM 차단 + 정상 working set 미만 시 재시작이 진단 신호로 수용 |
| Runtime tune | `MALLOC_TRIM_THRESHOLD_=131072` + `ARROW_DEFAULT_MEMORY_POOL=system` | glibc malloc free 시 OS 반환률 ↑, jemalloc → glibc 전환으로 RSS 추적성 ↑ |
| Close pattern | `with pq.ParquetWriter(...)` context manager + exception 시 tmp cleanup | atomic rename 전 실패 시 tmp 파일 누적 차단 + handle leak 가설 직접 fix |
| Observability scope (Phase 1) | 5 metric + Grafana dashboard (영구) — tracemalloc 은 별도 host-side script | tracemalloc 은 진단용 (12h × 10min snapshot), Prometheus 는 운영용 (5s sample) — 역할 분리 |
| GC interval | `MCTRADER_COMPACTOR_GC_INTERVAL_SECONDS=300` (interval-driven) | tick 마다 호출 시 CPU overhead, 5분 마다는 L3 cycle (1h+) 보다 짧아 retention 차단 충분 |
