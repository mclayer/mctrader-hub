# MCT-133: Compactor 런타임 안정화 — 32G mem_limit + glibc/arrow tune + ParquetWriter close audit

## §1 배경

MCT-132 Epic-A 의 A1 (Stabilize) 분기. `mctrader-compactor` 의 RAM 폭주 (48.5 GiB / 62.73 GiB) 에 대한 **즉시 OOM 차단** + **handle leak 가설 직접 fix**.

상위 Story: MCT-132. Spec: `docs/superpowers/specs/2026-05-11-compactor-memory-fix-design.md` §2 Epic-A A1.

## §2 목표 및 범위

- compose 레벨 cgroup `mem_limit: 32g` + `memswap_limit: 32g` (swap 사용 금지)
- 런타임 env: `MALLOC_TRIM_THRESHOLD_=131072` + `ARROW_DEFAULT_MEMORY_POOL=system` + `MCTRADER_COMPACTOR_GC_INTERVAL_SECONDS=300`
- L1/L2/L3 `pq.ParquetWriter` context manager wrap + exception 시 tmp 파일 정리
- runner `_tick()` 에 interval-driven `gc.collect()` 호출

## §4 설계 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| mem_limit value | 32 GiB | 호스트 62.73 GiB 의 절반 — 동일 호스트 다른 컨테이너 영향 차단 |
| memswap_limit | mem_limit 와 동일 | swap thrash 가 진단 노이즈 — swap 사용 금지 |
| MALLOC_TRIM_THRESHOLD_ | 131072 (128KB) | glibc malloc 이 free 후 OS 반환 trigger 임계, 기본보다 공격적 |
| ARROW_DEFAULT_MEMORY_POOL | system | jemalloc → glibc 전환, RSS 추적성 ↑ (PyArrow allocation 이 procfs RSS 에 반영) |
| GC interval | 300s | tick 마다 호출 시 CPU overhead, L3 cycle (1h+) 보다 짧아 retention 차단 충분 |
| Close pattern | `with pq.ParquetWriter(...)` context manager | PyArrow 가 `__enter__/__exit__` 지원 — try/finally 보다 간결, exception 보장 |
| Exception 시 tmp cleanup | atomic rename 전 실패 시 `tmp_path.unlink()` | tmp 누적 차단 — handle leak 가설 부수 효과 |
