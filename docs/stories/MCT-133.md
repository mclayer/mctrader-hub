# MCT-133: Compactor 런타임 안정화 — 32G mem_limit + glibc/arrow tune + ParquetWriter close audit

## §1 배경

MCT-132 Epic-A 의 A1 (Stabilize) 분기. `mctrader-compactor` 의 RAM 폭주 (48.5 GiB / 62.73 GiB) 에 대한 **즉시 OOM 차단** + **handle leak 가설 직접 fix**.

상위 Story: MCT-132. Spec: `docs/superpowers/specs/2026-05-11-compactor-memory-fix-design.md` §2 Epic-A A1.

## §2 목표 및 범위

- compose 레벨 cgroup `mem_limit: 32g` + `memswap_limit: 32g` (swap 사용 금지)
- 런타임 env: `MALLOC_TRIM_THRESHOLD_=131072` + `ARROW_DEFAULT_MEMORY_POOL=system` + `MCTRADER_COMPACTOR_GC_INTERVAL_SECONDS=300`
- L1/L2/L3 `pq.ParquetWriter` context manager wrap + exception 시 tmp 파일 정리
- runner `_tick()` 에 interval-driven `gc.collect()` 호출

## §4 설계 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| mem_limit value | 32 GiB | 호스트 62.73 GiB 의 절반 — 동일 호스트 다른 컨테이너 영향 차단 |
| memswap_limit | mem_limit 와 동일 | swap thrash 가 진단 노이즈 — swap 사용 금지 |
| MALLOC_TRIM_THRESHOLD_ | 131072 (128KB) | glibc malloc 이 free 후 OS 반환 trigger 임계, 기본보다 공격적 |
| ARROW_DEFAULT_MEMORY_POOL | system | jemalloc → glibc 전환, RSS 추적성 ↑ (PyArrow allocation 이 procfs RSS 에 반영) |
| GC interval | 300s | tick 마다 호출 시 CPU overhead, L3 cycle (1h+) 보다 짧아 retention 차단 충분 |
| Close pattern | `with pq.ParquetWriter(...)` context manager | PyArrow 가 `__enter__/__exit__` 지원 — try/finally 보다 간결, exception 보장 |
| Exception 시 tmp cleanup | atomic rename 전 실패 시 `tmp_path.unlink()` | tmp 누적 차단 — handle leak 가설 부수 효과 |

## §11 PMO 회고

**Retro pointer**: `docs/retros/RETRO-MCT-132-Compactor-Stabilize-Epic-A.md` (통합 RETRO)
**Perspective**: A1 런타임 안정화 — trade-off + ParquetWriter close pattern + gc interval 결정

### 잘된 점
- **Trade-off 명시 박제**: 32G mem_limit 선택이 호스트 62.73 GiB 의 절반 — "동일 호스트 다른 컨테이너 영향 차단" + "정상 working set 미만 시 재시작이 진단 신호" 의 dual rationale 가 §4 표에 박제. trade-off 미문서화로 인한 후속 의구심 차단.
- **ParquetWriter close pattern 일관성**: L1/L2/L3 3 tier 모두 동일하게 `with pq.ParquetWriter(...)` + `try/except tmp cleanup` 패턴 — 4 task (5/6) 가 일관 적용. mock target 만 변경하는 형태로 L2/L3 test 가 L1 test 의 copy-modify.
- **Interval-driven gc 의 적정 임계 선택**: 300s 가 L3 cycle (1h+) 대비 충분히 짧고 CPU overhead 무시 가능. test 는 `monkeypatch.setenv` 로 0.1s 로 축소해 2-tick test — interval 의 fuzz 없이 결정적 verification.
- **stdlib `gc` ↔ filesystem `gc` module 충돌 해소**: `from . import gc as _filesystem_gc` alias 패턴이 향후 다른 module 의 stdlib shadowing 회피 reference 로 활용 가능.
- **TDD 진정성 박제**: 4 신규 test (`test_l1_writer_close` / `test_l2_writer_close` / `test_l3_writer_close` / `test_runner_gc_interval`) 모두 FAIL → fix → PASS 패턴.

### 개선점
- **Real leak 가설 직접 검증 부재**: `with` context manager + tmp cleanup 이 handle leak 가설을 직접 fix 했으나, 24h baseline tracemalloc 손실로 인해 "이 fix 가 실제로 leak 을 잡았는가" 의 정량 검증은 deferred. 가설 → fix → 검증의 cycle 이 미완.
- **mem_limit 32G 가 hard cap — soft signal 부재**: 32G 도달 직전 (예: 28G) 의 early warning 없이 즉시 OOM-kill. `compactor_process_rss_bytes` 가 5s sample 이므로 Grafana alert (threshold ≥ 28G) 설정이 필요하나 본 Story scope 미포함.
- **GC interval 300s 의 결정 근거가 "L3 cycle 보다 짧음"** 으로 약함 — 실측 retention 패턴 (gen2 promotion 후 collect 까지 평균 시간) 측정 없이 heuristic 선택. tracemalloc baseline 의 retention 분포가 정량 근거가 됐어야 함.
- **`memswap_limit` = `mem_limit` 선택이 docker-compose v2 + v3 호환성 문제 잠재**: v3 schema 에서 `memswap_limit` deprecated → `deploy.resources.limits.memory` 권장. 본 Story 는 v2 compose 사용으로 OK 였으나 향후 v3 migration 시 점검 필요.

### 학습 자산
- **`with pq.ParquetWriter(...) as writer:` + tmp cleanup 패턴**: PyArrow writer 가 `__enter__/__exit__` 지원 — try/finally 보다 간결 + exception 보장. atomic rename 전 실패 시 `tmp_path.unlink()` 가 표준 cleanup. handle leak 가설 직접 fix 의 reference 구현.
- **`MALLOC_TRIM_THRESHOLD_=131072` + `ARROW_DEFAULT_MEMORY_POOL=system` 의 jemalloc → glibc 전환**: PyArrow allocation 이 procfs RSS 에 반영되도록 — RSS 추적성 ↑ 의 표준 tune. 향후 메모리 leak RCA Story 의 first-line 인프라.
- **stdlib module 과 동명 internal module 의 import alias 패턴**: `from . import gc as _filesystem_gc` + 외부 stdlib 는 `import gc` 그대로 — Python 의 absolute import 우선순위에 의존하지 않고 명시적 alias 로 안전.
- **interval-driven hook 의 test 패턴**: `monkeypatch.setenv` 로 interval 을 ms 단위로 축소 + 2-tick test — fuzz/sleep 없이 결정적 검증.
