### 2. codeforge-test MAJOR bump 처리 관찰

`codeforge-test 0.2.0 → 1.0.0` 은 본 세션 첫 **MAJOR** bump. 처리 lane:

- **이전 deprecation lane**: `codeforge-test` 가 한때 DEPRECATED 상태였음 (TestAgent/StatefulTestAgent spawn 불가)
- **REVIVED 결정**: ADR-055 (CFP-367) 로 IntegrationTestAgent (Sonnet) 전용 plugin 으로 부활
- **CLAUDE.md 처리**: §4 decider 결정 — "MAJOR 전면 재작성" 채택. 단순 버전 번호만 갱신하지 않고 `IntegrationTestAgent` / `§8.6` / `test-verdict-v2` 명시적 marker 3개 모두 plugin annotation 에 포함
- **Story workflow phase 추가**: `통합테스트 (IntegrationTestAgent, ADR-055, §8.6, test-verdict-v2)` 가 CI 테스트 ↔ 보안-테스트 사이에 신설 — phase 순서 자체가 변경됨
- **side effect**: `codeforge-develop 0.4.0` 이 동반 sibling 으로 `presets/docker-compose.test.yml` 신설 (CFP-367 sibling) — IntegrationTestAgent 가 의존하는 docker preset
- **잔존 위험**: **6 sister repo 중 어느 곳이라도 deprecated TestAgent/StatefulTestAgent 를 spawn 하는 Story phase / agent / hook / preflight script 가 남아있을 가능성** (특히 mctrader-engine / mctrader-data 의 CI 스크립트)

→ MAJOR bump 처리 templates: ① decider 가 "전면 재작성" 으로 명시 박제, ② plugin annotation 에 신규 contract 명 (test-verdict-v2 등) 명시, ③ Story workflow phase 순서 변경 박제, ④ sibling plugin (develop 0.4.0) 동반 변경 명시.

### 3. 4 종 동시 업그레이드 패턴

같은 날 4 plugin 동시 release 를 1 Story 로 묶은 lane 의 trade-off:

**장점** (본 Story 채택):
- CLAUDE.md 7개 섹션 한번에 정합 (plugin list 한번에 4개 annotation, decider 1회)
- `installed_plugins.json` 단일 commit
- 의무 플로우 (`feedback_codeforge_upgrade_process.md`) 가 4번 분리 실행 시 발생할 cognitive overhead 회피
- doc-only fast-pass 적용 가능 (review/test/security lane skip)

**위험** (감수):
- bisect 어려움 — 추후 plugin 충돌 시 어느 plugin 변경이 trigger 인지 commit 단위로 격리 안 됨 (`1f083ca` 한 commit 에 4 plugin 변경 합산)
- decider 1회로 4 결정 묶음 — review-verdict v4 / IntegrationTestAgent / ADR-055 / docker-compose preset 각각의 독립 결정점이 §4 3-row 표로 압축됨

→ **권고**: 4종 동시 + doc-only + 단일 release 날짜 (2026-05-10) → 1 Story 묶음 정당. 단, **추후 동일 패턴 재발 시 commit 분리** (4 plugin × 4 commit) 검토. bisect cost 가 cognitive cost 를 능가하는 시점 = MAJOR bump 가 2개 이상 동시 발생할 때.
