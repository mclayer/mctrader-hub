## 2. 도메인 해석 (Domain deputy 산출 통합)

### 2.1 Bithumb 공개 데이터 surface inventory + mctrader fit 평가

mctrader 가 현재 활용 중인 surface 와 미활용 surface 를 전수 inventory:

#### 활용 중 (in-use)
- **WS `transaction`** (T2 tick.v1) — MCT-63/MCT-65 채택 완료
- **WS `orderbookdepth`** (T3 orderbook.v1, L2 delta event stream) — MCT-63/MCT-65 채택 완료
- **REST `/public/candlestick/{order_currency}_{payment_currency}/{tick_type}`** (T1 ohlcv.v1) — MCT-9/MCT-12 채택 완료

#### 미활용 (gap)
| Surface | 종류 | mctrader fit 평가 |
|---|---|---|
| WS `ticker` (multi-window 30M/1H/12H/24H/MID push) | derived stream | **거부** — T1 (`ohlcv.v1`) 에서 derivable + ADR-005 lookahead trap 위험 (server-side rolling window 의 `available_from_ts` 모호) |
| WS `orderbooksnapshot` (L2 30-level full snapshot) | snapshot stream | **채택 P2** — §D11.6 fail-closed 의 "missing baseline" gap window 단축. 현재는 reconnect 시 첫 snapshot delta 도착 시점까지 reconstruction 불가 — full snapshot push 구독으로 baseline 신뢰성 ↑ |
| REST `/public/ticker/{order_currency}_{payment_currency}` | snapshot poll | **거부** — D1 동형 + polling stale + WS ticker 와 동일 lookahead trap |
| REST `/public/transaction_history/{order_currency}_{payment_currency}` (직전 ≤100건) | poll | **거부** — §D12.2 forward-only invariant 위반 (Bithumb 가 historical replay 미제공, 직전 ≤100건 polling = collector 가동 후 Already WS 수신 분과 중복 + race) |
| REST `/public/orderbook/{order_currency}_{payment_currency}` | snapshot poll | **거부** — WS `orderbooksnapshot` (P2 채택) 으로 cover, polling 추가 ROI 없음 |
| REST `/public/assetsstatus*` (입출금 상태) | static | **defer** — Live executor Epic 진입 시 검토 (현재 backtest 무관) |
| REST `/public/network-info` | static | **defer** — Live executor Epic |
| REST `/public/withdraw/minimum/{currency}` | static | **defer** — Live executor Epic |
| Private REST + Private WS (KRW 스팟) | account/order | **out-of-scope** — Bithumb KRW 스팟 미공개 + Live mode 별도 Epic |

#### Gap 신규 발견 (P1 채택의 진짜 motivation)
| Surface | mctrader fit 평가 |
|---|---|
| **거래소 metadata** (symbol list + tick_size + min_order_qty + fee_rate + asset_status snapshot) | **채택 P1** — ADR-002 H5 `ExchangeCapabilities` frozen dataclass 가 mctrader-engine 코드 grep 결과 **어떤 형태로도 미구현** (hardcode 도 부재). Live executor 진입 prereq blocker. 본 Story 가 H5 implementation seal 의 첫 채움. |

### 2.2 L3 미노출 발견 (ADR amendment trigger)

Bithumb KRW 스팟 public WS = **L2 30-level only**. ADR-009 §D9 ("L3 depth-ladder snapshot 예약, 미구현") 의 reservation 은 미래 거래소 (L3 노출 거래소 대상) 용으로 명시되어야 함 — Bithumb 한정으로는 §D9 활성화 불가. ArchitectPL 의 ADR-009 §D9 amendment 책임으로 carry-over.

### 2.3 채택 motivation 요약

본 Story 의 채택 = **2건만**:

1. **P1 `exchange_metadata.v1`**: ADR-002 H5 미구현 발견 → Live executor blocker 해소. mctrader 가 처음으로 Bithumb 거래소 capability (tick_size / min_order_qty / fee_rate / asset_status) 를 ADR-002 contract 로 보유.
2. **P2 WS `orderbooksnapshot`**: §D11.6 fail-closed 의 missing baseline window 단축. 기존 `orderbookdepth` 구독에 snapshot 채널 보강.

거부 surface 6건 (WS `ticker` / REST `/public/ticker` / REST `transaction_history` / REST `/public/orderbook` / `assetsstatus` / Private API) 은 §3 audit 표 + §10 Out-of-scope 박제.

C1 lock-in **Phase 1.5 amendment (2026-05-09)** — Phase 2 entry 직전 wiretap 측정 결과 박제:
- P1 `exchange_metadata.v1`: ~1.8 MB/yr 2-node, 영구 retention
- P2 `orderbook_snapshot.v1`: ~3 TB/yr 2-node uncapped (1-sec subsample throttle 후) → **180-day rolling ~1.5 TB** 2-node
- Wiretap 측정: BTC_KRW push interval ≈ 200ms (3 message 0.43s) — ArchitectPL 30s 가정 대비 ~150x. Throttle hard cap = 1-sec/sym (§D14.10).
- Phase 3 entry 시점 (50-sym × 7-day) 측정 후 (a) 50-sym 평균 native push interval (b) throttle drop ratio (c) zstd 압축률 (d) reconstruction baseline gap distribution → throttle 5-sec 강화 또는 retention 단축 (180→90) 결정. ADR-009 §D14.8 + §D14.10 amendment.

## 7. 보안 설계 (ArchitectPL 작성)
