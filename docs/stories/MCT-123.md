# MCT-123: Prometheus + Grafana 모니터링 도입

- **Status**: 완료 (hub commit `3b62be6` main 직접 push, PR #29 MERGED, PR #47 MERGED — 모두 2026-05-09)
- **Trigger**: mctrader 6 repo 운영 가시성 (operational visibility) 부재 — collector / runner / engine 의 처리량·지연·에러 누적 metric 수집 표준 부재
- **Repos**:
  - `mctrader-hub` — compose 모니터링 stack 5 서비스 + prometheus.yml scrape config + grafana provisioning

## 1. 배경

mctrader 6 repo 가 컨테이너화 (MCT-98 Epic, 2026-05-08) 완료되며 운영 환경에 진입. 그러나:
- collector / runner / engine 어느 쪽도 metric endpoint 노출 안 됨
- DB / 컨테이너 / 호스트 메트릭 수집 부재
- 운영 dashboard 없음 — 장애 발생 시 docker logs 수동 grep 의존
- Strategy 운용 시 처리량·지연 trend 가시화 수단 부재

Prometheus pull-based scrape 모델 + Grafana dashboard 를 hub compose 에 추가하고, mctrader-data + mctrader-engine 양쪽에 `/metrics` endpoint 노출.

## 3. 핵심 설계 결정

- **Pull 모델**: Prometheus 가 각 컨테이너 `/metrics` 를 scrape — collector/engine 측 push 코드 불필요, 백프레셔 없음
- **0.0.0.0 바인딩 변경**: HealthServer 가 `127.0.0.1` 로만 바인딩하던 기존 동작은 같은 컨테이너 내부 health check 만 고려 — Docker network 외부에서 Prometheus scrape 하려면 `0.0.0.0` 바인딩 필수. 보안상 컨테이너 외부 접근은 compose network 정책이 차단
- **Metric 명명 규칙**: `<component>_<subject>_<unit_or_total>` — `ingester_events_total`, `engine_open_positions` 등. Prometheus 표준 (snake_case + `_total` 카운터 접미)
- **5 서비스 스택 일괄 도입**: prometheus + grafana + postgres_exporter + cadvisor + node_exporter — DB / 컨테이너 / 호스트 메트릭까지 한 번에 가시화. 부분 도입 시 dashboard 작성 시 누락 메트릭 발견되어 재방문 비용 발생
