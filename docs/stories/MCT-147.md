## §2 도메인 해석 (DomainAgent 통합)

### 내부 제약 (mctrader 6-repo 도메인 해석)

**관찰 1 — `MCTRADER_DATA_ROOT` SSOT (ADR-009 §D12.1)**: mctrader-data 의 storage path SSOT 는 `MCTRADER_DATA_ROOT` 환경변수 + Docker named volume `mctrader_data`. 본 Story 의 NAS MinIO 는 이 SSOT 의 **cold tier extension** (L2/L3) — hot path (collector / WAL / L1) 는 local SSOT 유지.

**관찰 2 — 네트워크 파일시스템 우회 정합 (ADR-016 §A4 + ADR-017 §3)**: NFS/SMB/WebDAV 등 네트워크 파일시스템 위 named volume 사용 **금지** (POSIX fsync 보장 부재 → zero-loss invariant 위반). 본 Story 는 NAS 위에 MinIO 를 **직접 deploy** 함으로써 이 제약을 우회 — MinIO S3 API 는 application-layer object store 이므로 POSIX semantic 의존 없음.

**관찰 3 — bucket 이름 하드코딩 매칭 (mctrader-data/src/mctrader_data/compactor/minio_uploader.py)**: 기존 compactor 의 `BUCKET_NAME='mctrader-market'` 하드코딩과 본 Story 의 bucket 이름이 정확히 일치 — MCT-155 cutover 시 코드 변경 없이 endpoint 만 swap 가능.

**관찰 4 — ADR-027 reservation 정합**: D1~D11 결정 박제 → 본 Story 는 D1 (bucket layout) + D2 (TLS/auth) 의 implementation 만 author. 나머지 결정 (D3 distributed mode / D4 erasure coding 등) 은 Stage 2 (MCT-149+) scope.

### 도메인 용어 정합

- **cold tier**: L2 (compacted Parquet, 1 hour aggregation) + L3 (long-term archive, 1 day+ aggregation). hot tier (L1, WAL → 5 min Parquet) 와 storage backend 분리.
- **bucket layout**: 단일 bucket `mctrader-market` + Hive prefix (`exchange=BITHUMB/symbol=BTC_KRW/year=2026/month=05/day=12/...`) — S3 list/scan 효율 + Polars/DuckDB partition pruning 호환.
- **endpoint cutover**: mctrader-data 의 minio_uploader.py 가 local minio container endpoint → NAS minio endpoint 로 swap (MCT-155). 본 Story 는 NAS endpoint 만 가용화.

**Prerequisite 2 — NAS 측 btrfs/ext4 볼륨 가용**: 사용자 manual prerequisite. Synology 의 default file system (btrfs 권장 — checksum + snapshot 지원) 가용 확인. 예시 path: `/volume1/minio_data` (사용자 NAS 환경별 변경 가능).

**Prerequisite 4 — Container Manager Docker Compose import**: DSM 7.2 부터 native UI 지원. `docker-compose.yml` 파일을 Container Manager 의 "Project" 메뉴에서 import → 자동 deploy. CLI 접근 (SSH) 도 fallback 옵션.

**구현 가능성**: HIGH — MinIO single-node mode 는 single drive 만 요구하므로 NAS 환경에서 가장 단순한 deploy 형태. distributed mode 는 Stage 2 (MCT-149+) scope, 본 Story 비대상.

## §5 Acceptance Criteria

### AC (Given/When/Then 형식)

- **AC-1 (Bucket 자동 생성)**: GIVEN Synology Container Manager + btrfs volume 준비됨, WHEN `docker-compose up -d` 실행, THEN MinIO 컨테이너가 healthy 상태로 부팅 + `mctrader-market` bucket 자동 생성 (minio-init container 또는 mc client one-shot).
- **AC-2 (TLS handshake)**: GIVEN MinIO 부팅 완료, WHEN 외부 호스트에서 `curl https://NAS_IP:9000/minio/health/live` 호출, THEN HTTP 200 + TLS handshake 성공 (self-signed cert 의 경우 client 측 CA trust 박제).
- **AC-3 (재시작 idempotent)**: GIVEN MinIO 부팅 완료 + 임의 object PUT 박제, WHEN `docker compose down && docker compose up -d` 재시작, THEN bucket + data 모두 보존 (NAS local volume 무손실, named volume 정합).
- **AC-4 (Credential 평문 노출 0)**: GIVEN credential 평문 노출 검증, WHEN 작업 트리 + commit 에 access/secret key grep, THEN 0 matches (.env 가 `.gitignore` 박제, compose YAML 에 `${MINIO_ROOT_USER}` 변수 reference 만).
- **AC-5 (Runbook dry-run 가능)**: GIVEN runbook 2종 작성 완료, WHEN dry-run 수행 (deploy step 1~N, rotation step 1~M), THEN 각 step 이 실행 가능한 명령 + 예상 출력 박제 (step-by-step + idempotent 검증).

### Edge cases

- **EC-1 (NAS down 시 fallback)**: NAS 가 다운되어 네트워크 차단된 경우 — runbook 이 fallback 절차 명시 (MINIO_ENDPOINT local 복귀, mctrader-data 측 env override). **본 Story 는 문서화만**, 자동 fallback 은 MCT-155 scope.
- **EC-2 (TLS 인증서 만료)**: TLS cert 가 만료되는 경우 — rotation runbook 이 cert 갱신 절차 포함 (self-signed: openssl 재발급 / Let's Encrypt: Synology native renewal 검증).
- **EC-3 (Container Manager 가 Docker Compose v2 syntax 미지원)**: DSM 버전 < 7.2 인 경우 — runbook prerequisite section 에 명시 (사용자 NAS 업그레이드 의무).
- **EC-4 (btrfs 볼륨 full)**: NAS 측 디스크 full 발생 — MinIO 가 PUT 거부 (5xx). 본 Story 는 monitoring 미포함 (Stage 2 scope), runbook 에 수동 capacity 확인 절차만 명시.

### Non-goals (본 Story 비대상)

- **PoC 검증 (5종 항목)**: MCT-148 scope (throughput / latency / consistency / failure recovery / cost).
- **mctrader-data endpoint cutover**: MCT-155 scope (compactor minio_uploader.py 의 endpoint swap + production validation).
- **Distributed mode / Erasure coding**: Stage 2 (MCT-149+) scope (ADR-027 D3/D4 박제 예정).
- **자동 fallback (NAS down → local)**: MCT-155 scope (env override + health check automation).
- **Monitoring / Alerting**: Stage 2 scope (Prometheus exporter + alert rule).

### 사용자 확인 필요 (blocking — ArchitectPLAgent 진입 전 해소 의무)

- **Q1 (Issue 번호 박제)**: 본 Story 의 GitHub Issue 번호. Orchestrator 의 story-init.yml Action 이 생성 후 frontmatter `story_issues[0].number` + § header `Issue: TBD` 자리 swap. 본 Story file 의 frontmatter `number: TBD` + § header `Issue: TBD` 둘 다 후속 작업.
- **Q2 (NAS path 예시값)**: runbook 의 NAS local btrfs volume path 예시 (`/volume1/minio_data`) 가 사용자 NAS 환경과 일치하는지 확인. 다른 path 사용 시 runbook 의 path placeholder 일관성 유지.
- **Q3 (TLS 인증서 선택 default)**: self-signed vs Let's Encrypt 둘 다 runbook 지원이나, default option 선호 결정 (production-ready 권장 = Let's Encrypt, 빠른 setup = self-signed). Architect 단계에서 runbook 의 "권장 옵션" section 작성 시 반영.

## §6 Design (Change Plan 본문)

### §6.0 D2 결정점 amend — TLS → HTTP (Stage 1 한정)

ADR-027 reservation marker §D2 의 원안은 "TLS 강제 (self-signed OK)" 였으나, **사용자 결정 (2026-05-12, 본 Story Q3 답변)** 으로 **Stage 1 한정 HTTP 운영** 으로 amend.

**Amend rationale**:
- **운영 환경**: LAN 내부망 only (외부 노출 0, NAS 측 방화벽 port 9000/9001 이 mctrader 호스트 IP 만 allow)
- **Stage 1 scope**: feasibility spike — TLS handshake validation 은 본 Story 의 핵심 검증 항목 아님 (bucket bootstrap + idempotent restart 가 핵심)
- **setup 비용**: self-signed cert 의 client 측 CA trust 박제 + Let's Encrypt 의 public DNS 의존성 = Stage 1 의 spike 속도 저해
- **Stage 2 escalation 의무**: MCT-149 (ADR-027 본문 publish) 진입 시 **D2 wording 갱신 의무** — "Stage 1 = HTTP (LAN 내부망), Stage 2 = TLS 옵션 재검토 (사용자 confirm 필요)". 본 amend 의 ADR-027 reservation marker append 는 별도 Story 액션 또는 MCT-149 본문 author 시 일괄 반영.

**잔존 보안 면 (SecurityArch 박제)**:
- **노출 위험**: LAN sniffing 시 credential (MinIO root) + S3 PUT/GET payload 평문 전송
- **Mitigation**:
  - `.env` 파일 0600 권한 + `.gitignore` 박제 (commit 진입 차단)
  - 90d credential rotation runbook (MCT-147 산출물)
  - NAS 측 방화벽 port 9000/9001 = mctrader 호스트 IP only (LAN 내부망 전제 의무)
  - root account 외 별도 IAM user 분리 (Stage 2 scope, 본 Story 비대상)
- **AC-2 / NFR-2 갱신**: TLS handshake 검증 → **HTTP 200 health check** 로 대체 (`curl http://NAS_IP:9000/minio/health/live`)

### §6.2 bucket 초기화 — minio-init container (one-shot)

**Image**: `minio/mc:latest`

**depends_on**: `minio: { condition: service_healthy }`

**Command** (entrypoint override):
```sh
mc alias set local http://minio:9000 ${MINIO_ROOT_USER} ${MINIO_ROOT_PASSWORD}
mc mb --ignore-existing local/mctrader-market
mc anonymous set none local/mctrader-market
```

**Restart policy**: `"no"` (one-shot, bucket idempotent 생성 후 exit 0)

**Environment**: `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD` (.env 에서 주입, minio service 와 동일)

**Idempotent 보장**: `mc mb --ignore-existing` 플래그 + `mc anonymous set none` 은 이미 적용된 상태에서 재실행해도 변경 0.

1. **Step 1**: NAS DSM 7.2+ 확인 (`Control Panel → System → DSM Update`) + Container Manager package 설치 (`Package Center → Container Manager → Install`)
2. **Step 2**: SSH or File Station 으로 `/volume1/docker/minio/data` 디렉터리 생성 + 권한 설정 (`chown 1000:1000 /volume1/docker/minio/data && chmod 755 /volume1/docker/minio/data`)

6. **Step 6**: Console UI 접근 (`http://NAS_IP:9001`) → bucket `mctrader-market` 존재 확인 (minio-init 자동 생성)
7. **Step 7**: mctrader-data 측 endpoint 변경 (`MCT-155 scope`, runbook 만 명시 — 본 Story 는 변경 0)

- mctrader-data 측 `MINIO_ENDPOINT` env override 절차 (local minio container 복귀)
- 본 Story 는 **문서만** — 자동 fallback 은 MCT-155 scope

**부록 — Container Manager Compose Import 실패 시**:

- Compose v2 syntax 미지원 → SSH 접근 후 `docker compose -f /volume1/docker/minio/docker-compose.yml up -d` CLI fallback

### §6.5 docs/runbooks/nas-minio-secret-rotation.md 구조

**Cadence**: 90d (예: 2026-08-10 → 2026-11-08), audit log 박제 의무

**Step-by-step**:

### §7.3 SecurityArchitectAgent — HTTP 운영 trade-off

**위험 분석**:
- LAN sniffing 시 credential (MinIO root user/password) + S3 PUT/GET payload 평문 전송 위험

**Mitigation (본 Story 적용)**:

- NAS 측 방화벽 port 9000/9001 access control = mctrader 호스트 IP only (runbook prerequisite 박제)

**Stage 2 escalation 의무**:
- MCT-149 (ADR-027 본문 publish) 진입 시 TLS 활성화 재검토 의무 박제 (D2 wording 갱신)
- 사용자 confirm 후 Let's Encrypt or self-signed cert 선택

**D2 amend rationale**: 본 Story §6.0 참조 (Stage 1 한정 HTTP, Stage 2 TLS 재검토 의무).

### §7.4 OperationalRiskArchitectAgent — 운영 리스크 register

**Risk 1 — NAS 정전**:
- 영향: MinIO down → mctrader-data compactor 의 S3 PUT 실패 → retry queue 누적
- Mitigation: Stage 1 = 문서만 (MCT-150 scope 의 retry queue 설계). 본 Story 의 runbook 에 NAS 복귀 후 retry queue drain 절차 명시.

**Risk 2 — Container Manager 재시작 / NAS 재부팅**:
- 영향: MinIO container 자동 부팅 의무 (downtime 최소화)
- Mitigation: `restart: unless-stopped` 정책 + healthcheck 30s interval. Container Manager 의 auto-start 옵션 활성화 (runbook step 4 박제).

**Risk 3 — 볼륨 mount 실패**:
- 영향: `/volume1/docker/minio/data` 디렉터리 부재 또는 권한 오류 시 MinIO startup 실패
- Mitigation: runbook step 2 의 디렉터리 생성 + 권한 설정 의무 + healthcheck FAIL 시 docker-compose 알림 (runbook step 5 verify)

**Risk 4 — btrfs/ext4 디스크 full**:
- 영향: MinIO PUT 5xx 거부 → mctrader-data compactor 실패
- Mitigation: Stage 1 = monitoring 미포함 (Stage 2 Prometheus exporter scope). 본 Story 의 runbook 부록 에 수동 capacity 확인 절차 명시 (Container Manager UI 에서 NAS 디스크 사용량 확인).

**Risk 5 — Container Manager Compose v2 미지원 (DSM < 7.2)**:
