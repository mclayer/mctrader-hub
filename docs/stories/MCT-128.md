### 3. 잠재 risk 가 실제로는 zero exposure 였던 lane 관찰

본 Story 의 가장 큰 lane 발견:

- **사전 가정**: codeforge-test 1.0.0 에서 spawn 불가가 된 deprecated agent (TestAgent / StatefulTestAgent) 가 6 sister repo 의 `.github/workflows/`, `scripts/`, agent override, hook 어딘가에 활성 spawn 으로 잔존할 가능성 (RETRO-MCT-127 §5 (HIGH))
- **실제**: 6-repo grep 결과 활성 spawn 호출 **0건**. mctrader-hub 의 `run-tests.sh` / `run-perf.sh` 주석 2개에만 historical reference 잔존 (실행 영향 0)
- **해석**: codeforge-test deprecation lane (TestAgent → IntegrationTestAgent) 이 plugin 차원에서 차단됐고, mctrader 6-repo 가 plugin 의 agent contract 만 의존했기 때문에 silent fail risk 가 실제로는 발생 불가 — plugin 의 `agents/` 경로에 deprecated agent 가 존재하지 않으므로 spawn 시도 자체가 plugin loader 단계에서 차단

→ **MAJOR bump risk assessment lane 박제**:
- ① upstream plugin 의 agent removal 은 consumer spawn 시도 시 plugin loader 차단 → silent fail 가능성 zero
- ② consumer 잔존 risk 는 **comment / docstring / prompt 내 historical reference** 에 한정 — 실행 contract 와 무관
- ③ 단, IDE/agent prompt 가 historical reference 를 보고 deprecated agent 를 호출하려 시도하면 사용자 confusion 발생 가능 — 본 Story 의 주석 갱신은 그 lane 차단

본 결과는 향후 동일 패턴 (plugin agent removal / contract version bump / agent rename) 발생 시 **사전 risk 평가 단축** 가능: ① plugin loader 차단 여부 확인 → ② consumer comment / docstring grep → ③ historical reference 만 갱신 (실행 contract 변경 없음).
