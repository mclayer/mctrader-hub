## 2. 문제 진단 (2026-05-09 세션 실측)

### 2.1 현재 손상 현황

- 총 Parquet 파일 807개 중 **299개 손상** (Parquet magic bytes 없음)
- 손상 원인: `TickWriter`/`OrderBookWriter` 가 `close()` 전에 프로세스 종료 → footer 미기록
- `TickWriter.batch_size=500` → SIGKILL 시 최대 499건 버퍼 유실
- `SIGTERM` 핸들러 미등록 → `docker stop`/`compose restart` 도 즉시 종료 (chore PR #17로 SIGTERM은 수정됨)

### 2.2 현재 아키텍처 한계

```
WS(bithumb) → CollectorDaemon → TickWriter(buffer=500) → Parquet
                                                         ↑ footer 없으면 전체 파일 손상
```

- 단일 컨테이너가 WS 수신 + 버퍼링 + Parquet 쓰기 모두 담당
- 다른 exchange 추가 시 코드 분기 or 컨테이너 복제 필요
- SIGKILL/OOM 시 버퍼 데이터 완전 유실

## 3. 요구사항

### 3.1 내성 요구사항

| 시나리오 | 현재 | 목표 |
|---|---|---|
| SIGTERM (docker stop) | 파일 손상 | 0건 유실 (chore PR #17) |
| SIGKILL / OOM kill | 최대 499건 유실 | **0건 유실** |
| 프로세스 crash | 최대 499건 유실 | **0건 유실** |
| 재시작 후 재기동 | 손상 파일 잔존 | 자동 복구 |

### 3.2 확장성 요구사항

- 새 exchange 추가 = **Ingester 컨테이너 1개 추가**만으로 충분
- Compactor는 exchange-agnostic
- 기존 Parquet 경로 (`market/ticks/schema_version=tick.v1/...`) 유지

### 3.3 채널 커버리지

- `transactions` (tick), `orderbookdepth`, `orderbooksnapshot` — 3채널 전체

## 4. 목표 아키텍처 (사용자 확인)

```
[bithumb-ingester]  ──┐
[upbit-ingester]    ──┤──→ WAL volume (NDJSON) ──→ [compactor]
[binance-ingester]  ──┘         ↑ per-exchange                │
                                  per-symbol                   ├──→ L1 (5-min Parquet)
                                  per-channel                  ├──→ L2 (1-hour Parquet)
                                  per-date                     └──→ L3 (1-day Parquet)
```

### 계층형 compaction (사용자 지정)

| Level | 입력 | 출력 | 주기 |
|---|---|---|---|
| WAL→L1 | NDJSON WAL segment | 5분 단위 Parquet | 5분마다 sealed segment |
| L1→L2 | 5분 Parquet × 12 | 1시간 Parquet | 매 시 정각 |
| L2→L3 | 1시간 Parquet × 24 | 1일 Parquet | UTC 자정 |
