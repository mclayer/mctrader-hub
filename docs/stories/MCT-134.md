# MCT-134: Compactor 관측 인프라 — /metrics HTTP server + 5 Prometheus metrics + Grafana dashboard

## §1 배경

MCT-132 Epic-A 의 A2 (Observability) 분기. compactor 의 메모리 동작을 **운영 시 시계열 관측** 가능하게 해 Epic-B (v2 재설계) 진단 근거 확보.

기존 compactor 는 `/metrics` endpoint 자체가 부재 (`mctrader-data-compactor` Prometheus scrape job 은 존재하나 응답 없음 상태).

상위 Story: MCT-132. Spec: `docs/superpowers/specs/2026-05-11-compactor-memory-fix-design.md` §2 Epic-A A2.

## §2 목표 및 범위

- Grafana dashboard "Compactor Memory & Throughput" (5 panels)

## §4 설계 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| HTTP server lib | `prometheus_client.start_http_server` | Standard, daemon thread 자동 관리, 별도 ASGI app 도입 회피 |
| Observer thread interval | 5초 | PyArrow / GC stats 가 5s sample 로 충분, scrape interval (보통 15s) 보다 빠름 |
| `compactor_process_rss_bytes` 측정 | Linux `/proc/self/status` VmRSS + macOS `resource.getrusage` + Windows ctypes/psapi `GetProcessMemoryInfo` | 3-platform fallback — Windows 개발 환경 + 운영 Linux 컨테이너 모두 지원 |
| `tier_pending_segments` semantics | L1=실제 sealed backlog 수, L2/L3 = stale interval count (`max(0, int((now - _last_*) / interval))`) | 정확 계산은 별도 Story scope, Phase 1 minimal — last_processed delta 로 대체 |
| tracemalloc collector | host-side script `docker cp` + `docker exec` attach | compactor 코드에 tracemalloc 내장 시 prod overhead — 진단 도구 분리 |
| Grafana provisioning path | `/etc/grafana/provisioning/dashboards` | MCT-123 의 기존 path (`/var/lib/grafana/dashboards`) 가 bind mount 없는 broken — 동반 fix |

## §11 PMO 회고

**Retro pointer**: `docs/retros/RETRO-MCT-132-Compactor-Stabilize-Epic-A.md` (통합 RETRO)
**Perspective**: A2 관측 인프라 — 진정성 + tier_pending epoch bug + dashboard 자기검증 게이트 부재

### 잘된 점
- **관측 인프라의 진정성**: 5 metric 이 모두 운영 의미 있는 신호 — `process_rss_bytes` (호스트 OOM threshold) / `pyarrow_total_allocated_bytes` (Arrow internal) / `python_gc_gen_count` (Python heap) / `tier_pending_segments` (compaction throughput vs ingestion rate) / `writer_open_count` (handle leak direct signal). 단순 cargo-cult metric 아님.
- **3-platform fallback 박제**: Linux procfs (`/proc/self/status` VmRSS) + macOS (`resource.getrusage`) + Windows (ctypes/psapi `GetProcessMemoryInfo`) — Windows 개발 환경 + 운영 Linux 컨테이너 모두 지원. test 가 platform-conditional skip 없이 통과.
- **TDD 가 운영 bug 즉석 catch**: Task 7 의 `_last_l2`/`_last_l3` epoch=0 nonsense (`(now - 0.0) / interval` ≈ 5.9M / 494K 거짓 값) 을 `test_runner_pending_init.py` 가 regression guard 로 박제. 운영자 panic 회피 + dashboard threshold-based alerting 무효화 회피의 dual value.
- **즉시 운영 시그널 발견**: Phase 1 land 직후 `tier_pending_segments{tier="L1"}=22,497` 노출 — observability 도입 즉시의 ROI 입증. Epic-B 의 B1 staging 진단 input 으로 직접 활용 가능.
- **tracemalloc 의 도구 분리 결정**: compactor 코드에 tracemalloc 내장 시 prod overhead — host-side script + `docker cp` + `docker exec` attach 패턴이 진단/운영 분리 정합.

### 개선점
- **dashboard.yml path 자기검증 게이트 부재 (CRITICAL pre-existing gap)**: MCT-123 의 dashboard provisioning path 가 `/var/lib/grafana/dashboards` (bind mount 없는 경로) 로 broken 상태였으나 **MCT-123 verification 단계에서 발견 못함** — 모든 Grafana dashboard 가 silent 하게 load 안 되고 있던 상태. Task 8 동반 fix 로 해소했으나, MCT-123 시점에 e2e dashboard load check 가 acceptance criteria 에 포함됐어야 함.
- **tier_pending semantics 의 epoch=0 nonsense**: `_last_l2`/`_last_l3` epoch=0 일 때 `(now - 0) / interval` 이 천문학적 stale count 노출 — 초기화 단계 의식 부족. fresh start 시 metric 노출 전 초기화 시퀀스의 일반 패턴 (guard epoch + prime labels with 0) 을 standard practice 로 박제 필요.
- **tier_pending L2/L3 semantics 의 stale interval count 가 임시 해결**: §4 표에서 "정확 계산은 별도 Story scope, Phase 1 minimal — last_processed delta 로 대체" 라 명시 — 운영자가 dashboard 해석 시 L1 (실제 backlog) vs L2/L3 (interval 추정) 의 semantics 차이를 알아야 함. Epic-B B1 시점에 통일 필요.
- **tracemalloc collector 의 docker cp + exec 패턴 의 fragility**: docker recreate 마다 collector reset — Phase 1 의 매 Task 마다 baseline 손실. dedicated sidecar container 또는 host-side mount 패턴 검토.

### 학습 자산
- **fresh start metric 초기화 패턴**: epoch=0 nonsense 회피를 위한 `if self._last_X > 0` guard + prime labels with 0. `test_runner_pending_init.py` 가 regression guard 의 reference 구현.
- **3-platform RSS 측정 패턴**: Linux procfs + macOS resource + Windows ctypes/psapi 의 fallback chain — 향후 다른 process metric (CPU, fd count) 도입 시 동일 패턴 적용 가능.
- **dashboard provisioning path 의 silent failure**: bind mount 없는 경로에 provisioning 파일이 있어도 Grafana 가 silent 하게 skip — 운영 검증 시 dashboard UI 의 dashboard list 응답 카운트로 e2e 검증 필요. MCT-123 부터 broken 이었으나 발견 못한 점이 cross-Story 회귀의 case study.
- **observability 도입의 즉시 ROI**: L1 backlog 22,497 segments 가 metric 노출 즉시 visible — "metric 부재 = bug 부재" 의 false security 회피의 case study.
- **HTTP server thread 의 daemon 패턴**: `prometheus_client.start_http_server` 가 daemon thread 자동 관리 — 별도 ASGI app 도입 회피의 minimal overhead 선택.
